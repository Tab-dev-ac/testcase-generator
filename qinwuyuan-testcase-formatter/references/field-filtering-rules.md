# 字段过滤规则

## 概述

本文档详细说明了在转换为秦务员格式时，哪些字段需要保留，哪些字段需要去除。

## 参考模板

根据 `/Users/tablee/Downloads/秦务员用例格式模板.md`，最终输出**只包含**以下字段：

```markdown
# 项目名：陕西秦务员

## 模块名：自动首页

### 用例名：常规版切换到长辈版测试

- 操作步骤1：点击'长辈版'按钮

- 验证：页面有"常规版"按钮

- 操作步骤2：点击'常规版'按钮

- 验证：'页面有"长辈版"按钮'
```

## 必须保留的字段 ✅

| 字段名 | 格式 | 示例 | 说明 |
|--------|------|------|------|
| 项目名 | `# 项目名：XXX` | `# 项目名：陕西秦务员` | 文档标题，标识项目 |
| 模块名 | `## 模块名：XXX` | `## 模块名：自动首页` | 二级标题，标识功能模块 |
| 用例名 | `### 用例名：XXX` | `### 用例名：常规版切换到长辈版测试` | 三级标题，标识测试用例 |
| 操作步骤 | `- 操作步骤N：XXX` | `- 操作步骤1：点击'长辈版'按钮` | 具体操作步骤 |
| 验证点 | `- 验证：XXX` | `- 验证：页面有"常规版"按钮` | 验证点 |

## 必须去除的字段 ❌

### 1. 优先级字段

**常见格式**：
```
- 优先级: P0
- 优先级: P1
- 优先级: P2
- 优先级：高/中/低
```

**处理方式**：完全去除，不保留在最终输出中

**示例**：

输入：
```markdown
#### TC-F01-001 - 列表页展示所有记录
- 优先级: P0
- 测试步骤:
  1. 进入订阅管理页面
```

输出：
```markdown
### 用例名：列表页展示所有记录

- 操作步骤1：进入订阅管理页面
```

---

### 2. 状态字段

**常见格式**：
```
- 状态: 待执行
- 状态: 通过
- 状态: 失败
- 状态: 阻塞
```

**处理方式**：完全去除，不保留在最终输出中

**示例**：

输入：
```markdown
#### TC-F01-001 - 测试用例
- 状态: 待执行
- 测试步骤:
  1. 执行测试
```

输出：
```markdown
### 用例名：测试用例

- 操作步骤1：执行测试
```

---

### 3. 测试用例编号字段

**常见格式**：
```
#### TC-F01-001 - 用例名
#### TC-BP-001 - 用例名
#### TC-001 - 用例名
```

**处理方式**：去除编号，只保留用例名称

**示例**：

输入：
```markdown
#### TC-F01-001 - 列表页展示所有记录
```

输出：
```markdown
### 用例名：列表页展示所有记录
```

---

### 4. 前置条件字段（特殊处理）

**处理原则**：
- 如果前置条件包含可执行的动作 → **转换为操作步骤1**
- 如果前置条件只是状态描述 → **直接去除**
- 如果前置条件信息不足 → **直接去除**

#### 情况1：可执行的前置条件 → 转换为步骤

输入：
```markdown
- 前置条件: 安全管理员登录系统
- 测试步骤:
  1. 进入订阅管理页面
```

输出：
```markdown
- 操作步骤1：安全管理员登录系统

- 操作步骤2：进入订阅管理页面
```

#### 情况2：多个可执行的前置条件

输入：
```markdown
- 前置条件: 已登录系统，拥有订阅管理权限
- 测试步骤:
  1. 点击新增按钮
```

输出：
```markdown
- 操作步骤1：登录系统

- 操作步骤2：确认拥有订阅管理权限

- 操作步骤3：点击新增按钮
```

#### 情况3：状态描述的前置条件 → 直接去除

输入：
```markdown
- 前置条件: 系统正常运行
- 前置条件: 数据已准备好
- 测试步骤:
  1. 执行测试
```

输出：
```markdown
- 操作步骤1：执行测试
```

#### 情况4：信息不足的前置条件 → 直接去除

输入：
```markdown
- 前置条件: 环境已就绪
- 测试步骤:
  1. 执行测试
```

输出：
```markdown
- 操作步骤1：执行测试
```

- 创建人: 张三
- 优先级: P0
- 状态: 待执行

## 字段过滤决策树

```
输入字段
    │
    ├─ 项目名、模块名、用例名
    │   └─ ✅ 保留
    │
    ├─ 操作步骤、验证点
    │   └─ ✅ 保留
    │
    ├─ 前置条件
    │   ├─ 包含可执行动作？
    │   │   ├─ 是 → ✅ 转换为操作步骤1
    │   │   └─ 否 → ❌ 去除
    │
    ├─ 优先级 (P0/P1/P2)
    │   └─ ❌ 去除
    │
    ├─ 状态 (待执行/通过/失败)
    │   └─ ❌ 去除
    │
    ├─ 用例编号 (TC-XXX)
    │   └─ ❌ 去除
    │
    └─ 其他元数据
        └─ ❌ 去除
```

## 前置条件处理详细规则

### 判断标准

**可执行的前置条件**：
- ✅ 包含动作动词（登录、进入、点击、输入、选择等）
- ✅ 明确的步骤描述
- ✅ 可以独立执行

**不可执行的前置条件**：
- ❌ 状态描述（系统正常、环境就绪）
- ❌ 抽象描述（数据已准备好、配置已设置）
- ❌ 缺少具体操作步骤

### 转换规则

1. **单个可执行前置条件**
   ```
   输入：- 前置条件: 用户已登录系统
   输出：- 操作步骤1：用户已登录系统
   ```

2. **多个前置条件**
   ```
   输入：- 前置条件: 已登录系统，拥有管理员权限，网络正常
   输出：
     - 操作步骤1：登录系统
     - 操作步骤2：确认拥有管理员权限
   （"网络正常"是状态描述，去除）
   ```

3. **复杂前置条件**
   ```
   输入：- 前置条件: 使用Chrome浏览器访问系统，分辨率1920x1080
   输出：
     - 操作步骤1：使用Chrome浏览器访问系统
     - 操作步骤2：设置分辨率为1920x1080
   ```

---

## 总结

**记住这5条规则**：

1. ✅ 只保留模板中出现的字段
2. ❌ 优先级、状态、编号必须去除
3. ❌ 其他元数据全部去除
4. 🔄 前置条件：可执行则转步骤，不可执行则去除
5. 🔄 步骤编号：重新从1开始编号

这样就能确保生成的文档完全符合秦务员格式规范！
